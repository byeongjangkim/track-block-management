# 선로차단작업 관리 프로그램 개발 계획서

**작성일:** 2026년 4월 3일
**대상 조직:** 한국철도공사 부산경남본부
**개발 도구:** Claude Code + VSCode 병행

---

## 1. 프로그램 개요

한국철도공사 부산경남본부 관내의 선로차단작업 승인 내역을 통합 관리하는 웹 기반 프로그램이다. 차단명령 승인문서의 디지털화, 철도노선도 기반 시각화, 작업 현황 조회 등의 기능을 제공한다.

---

## 2. 업무 프로세스

1. 선로차단작업 승인 내역 확인
2. 대상 차단명령 승인문서 출력 또는 PDF 변환
3. 문서 스캔 후 프로그램에 업로드
4. 차단명령을 철도노선도에 표시
5. 일자별 차단명령 집계표에서 조회
6. 차단작업 현황 관리

### 차단작업 현황 포함 항목
- 분야, 차단명령 종류
- 장비작업 및 인력작업 유무
- 외부작업(공사) 및 내부작업 구분
- 작업 관계자: 작업책임자, 안전관리자, 운행안전협의자, 열차감시원 등
- 안전관리 항목

## 4. 철도노선도 설계

### 4.1 기본 방침
- **GIS 데이터 기반** SVG 노선도 제작 (스캔 이미지 불필요)
- OpenStreetMap(Overpass API) 또는 VWORLD에서 철도 선형 데이터 취득
- 독자적인 **철도 전용 좌표계** 적용 (거리정 기준)

### 4.2 데이터 소스

| 소스 | 라이선스 | 형식 | 비고 |
|---|---|---|---|
| OpenStreetMap Overpass API | ODbL 1.0 (출처 표기) | GeoJSON | API 키 불필요, 즉시 사용 |
| VWORLD WFS (nsdi.go.kr) | 공공누리 1유형 | SHP/GeoJSON | API 키 무료 발급 필요 |

### 4.3 좌표계
- 기준: **노선명 + 거리정(km)**
- 방향: 시점(기점) → 종점 방향으로 km 단위 표시
- 상선/하선 구분: SVG y 오프셋으로 분리 표시
- 예시: `경부선 KP 325.4` → 경부선 325.4km 지점
- 방향 코드: `UP` (상선) / `DOWN` (하선)

### 4.4 노선도 제작 파이프라인

```
[1단계] 데이터 수집
  Overpass API → raw/osm_busan_gyeongnam.geojson
  (pipeline/download_osm.py + osmtogeojson)

[2단계] 노선별 GeoJSON 분리 (Python + geopandas)
  pipeline/extract_routes.py
  → processed/gyeongbu.geojson
  → processed/gyeongjeon.geojson
  → processed/donghae.geojson

[3단계] SVG 변환 (mapshaper)
  mapshaper processed/gyeongbu.geojson \
    -proj merc -simplify 0.5% \
    -o format=svg svg/gyeongbu.svg

[4단계] 거리정 앵커 포인트 매핑
  pipeline/anchor_editor.py --route gyeongbu
  → anchors/gyeongbu.json (km ↔ SVG 좌표 대응표)

[5단계] 시설물 JSON 작성
  → facilities/gyeongbu.json (역, 건널목, 과선교 등)

[6단계] 역구내 배선도 서브맵 (Phase 3)
  → station_maps/[역코드].svg

[7단계] 프론트엔드 배포
  pipeline/deploy.py → frontend/public/maps/gyeongbu.svg
```

### 4.5 표시 시설물

| 종류 | 코드 | 비고 |
|---|---|---|
| 기차역 | `STATION` | 역구내 배선도 연결 (Phase 3) |
| 철도건널목 | `CROSSING` | |
| 과선교 | `OVERPASS` | |
| 철도변전소 | `SUBSTATION` | |
| 터널 | `TUNNEL` | |
| 교량 | `BRIDGE` | |

### 4.6 역구내 배선도
- Phase 3에서 구현 — 스캔 또는 GIS 데이터로 별도 SVG 제작
- 본선, 측선 등 구분 표시
- 역 선택 시 서브맵(팝업)으로 표시

## 5. 주요 기능

### 5.1 캘린더 뷰
- 월간 캘린더에 날짜별 차단명령 건수 배지 표시
- 날짜 클릭 시 당일 차단명령 목록 + 노선도 오버레이 연동

### 5.2 노선도 시각화
- 차단구간을 노선도에 색상 오버레이로 표시
- 상선/하선 구분 표시
- 줌(wheel), 패닝(drag) 기능 — D3.js `zoom()` 사용
- 시설물 클릭 시 상세 정보 표시

### 5.3 문서 관리
- 승인문서 스캔본(PDF) 업로드 및 뷰어 (react-pdf)
- 문서 검색 및 조회
- PDF는 `backend/uploads/` 에 저장, DB에는 상대 경로만 기록

### 5.4 집계 및 조회
- 일자별 차단명령 집계표
- 분야별, 노선별 통계
- 상세 조회 기능

### 5.5 연동 기능 (Phase 3 이후)
- 기상데이터 연동 (공공데이터포털 API — data.go.kr)

## 6. 데이터 구조 (주요 테이블)

### 노선 (routes)
```
route_id, code(영문코드), name(노선명), start_km, end_km,
up_direction(상선방향명), down_direction(하선방향명)
```

### 시설물 (facilities)
```
facility_id, route_id,
type(STATION/CROSSING/OVERPASS/SUBSTATION/TUNNEL/BRIDGE),
km(거리정), name(명칭), has_station_map(역구내배선도여부)
```

### 거리정 앵커 포인트 (route_anchors)
```
anchor_id, route_id, km, svg_x, svg_y
```
*앵커 사이 구간은 선형 보간으로 SVG 좌표 계산*

### 차단명령 (block_orders)
```
block_order_id, route_id, direction(UP/DOWN),
start_km, end_km, work_date, start_time, end_time,
field(분야), block_type(차단종류),
has_equipment(장비작업), has_manpower(인력작업), is_external(외부/내부),
work_manager(작업책임자), safety_manager(안전관리자),
operation_safety(운행안전협의자), train_watcher(열차감시원),
safety_items(안전관리항목), document_path(PDF파일경로)
```

### 사용자 (users)
```
user_id, username, password(bcrypt해시), name,
role(admin/editor/viewer), is_active
```

## 7. 개발 단계 (Phase)

### Phase 1 — 기반 구축 (SQLite, M2 맥북)
- [ ] 대상 노선 목록 및 거리정 범위 확정 (도메인 담당자 확인)
- [ ] GIS 도구 설치: `brew install gdal`, `npm install -g mapshaper osmtogeojson`
- [ ] OSM Overpass API로 부산경남 철도 GeoJSON 수집 (download_osm.py)
- [ ] 노선별 GeoJSON 분리 (extract_routes.py + geopandas)
- [ ] mapshaper로 GeoJSON → SVG 변환 (1개 노선 시범)
- [ ] 거리정 앵커 포인트 매핑 (anchor_editor.py)
- [ ] DB 스키마 설계 및 Alembic 마이그레이션 (`database/`)
- [ ] 기본 웹 뷰어 구현 (줌, 패닝, 시설물 클릭)
- [ ] 차단명령 기본 CRUD API (FastAPI)

### Phase 2 — 핵심 기능 구현 (SQLite → PostgreSQL 전환)
- [ ] 로그인 및 JWT 사용자 인증
- [ ] 차단구간 노선도 시각화 (색상 오버레이)
- [ ] 캘린더 뷰 구현
- [ ] PDF 문서 업로드 및 뷰어
- [ ] PostgreSQL 전환 (`brew install postgresql@15`)

### Phase 3 — 고도화 (PostgreSQL)
- [ ] 역구내 배선도 연결
- [ ] 집계표 및 통계 기능
- [ ] 기상데이터 API 연동 (data.go.kr)
- [ ] 모바일 반응형 UI 최적화

### Phase 4 — 확장 (검토)
- [ ] 사내 Linux 서버 또는 클라우드 이전 (Caddy/Nginx 리버스 프록시)
- [ ] 알림 기능 (작업 일정 알림 등)
- [ ] 보고서 자동 생성

## 11. 서버 이전 절차 (Mac → Linux)

코드는 동일하며 `.env` 파일만 서버 환경에 맞게 새로 작성한다.

```bash
# Linux 서버에서
git clone [저장소URL] track-block-management
cd track-block-management

# 백엔드
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
nano .env          # DATABASE_URL, SECRET_KEY, UPLOAD_DIR 서버값 입력
alembic upgrade head
uvicorn app.main:app --host 0.0.0.0 --port 8000

# 프론트엔드 (서버에서는 빌드 후 정적 파일 서빙)
cd ../frontend
npm install
VITE_API_URL=http://서버IP:8000 npm run build
# dist/ 폴더를 Nginx 또는 Caddy로 서빙
```

| 항목 | Mac (개발) | Linux (운영) |
|---|---|---|
| `DATABASE_URL` | `sqlite:///./db.sqlite3` | `postgresql://localhost/track_block` |
| `UPLOAD_DIR` | `./uploads` | `/var/data/track-block/uploads` |
| 프론트 실행 | `npm run dev` | `npm run build` → Nginx 서빙 |
| 코드 변경 | 필요 없음 | 필요 없음 |
